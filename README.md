# bert-mm-bot

Autonomous Meteora DLMM market-making bot for the **BERT/SOL** pool on Solana mainnet, with an optional on-chain depositor vault and a Telegram control surface.

The bot keeps a single concentrated-liquidity position centered on the current price, rebalances when the position drifts out of range, and (when the vault is enabled) lets depositors put SOL or BERT into the pool in exchange for vault shares whose USD value tracks the LP's net asset value.

---

## At a glance

- **Venue:** Meteora DLMM, `0.10%` base-fee pool `4rkbxnvmXagghqoV59jGZRcRUu94HHHq7axvFz8ERGMh` (bin-step 10).
- **Pair:** BERT (`HgBRWfYxEfvPhtqkaeymCQtHCrKE46qQ43pKe8HCpump`) / SOL.
- **Chain:** Solana mainnet. RPC: Helius (primary) with public fallback.
- **Runtime:** Node 22 + TypeScript 5 + pnpm. Single systemd service (`bert-mm-bot.service`) running as user `bertmm`.
- **State:** `/var/lib/bert-mm-bot/state.db` (SQLite, via `better-sqlite3`). Hot wallet keyfile: `/etc/bert-mm-bot/hot-wallet.json` (root:bertmm 600).
- **Config:** `/etc/bert-mm-bot/config.yaml` (root:bertmm 640). Validated on load via zod.

## What the bot does

### Core market-maker (always on)

1. **Polls the pool every 30 s.** Pulls a trusted mid price from multiple independent sources (Jupiter swap quote + DexScreener) and requires at least two to agree within `oracleDivergenceBps` (default 1.5%) before acting.
2. **Holds one concentrated-liquidity position** centered on that mid, with configurable width (`rangeWidthPct`, capped by the venue — up to ~7% for a 70-bin max at bin-step 10).
3. **Rebalances** when the price has been *sustained* outside the range for `sustainedMinutes`, subject to every hard limit below before the rebalance is attempted.
4. **Collects fees** opportunistically: either on every rebalance (`feeCollectionMode: on_rebalance`) or on demand via the `collect-fees` CLI.
5. **Halts itself** on any of: oracle disagreement, repeated RPC failure, drawdown trip, low SOL balance, kill-switch file present, or `enabled: false` in config (re-read every tick for live pause/resume).

It is intentionally a single-position passive market maker — not a quoting bot, not an arbitrageur, not a directional strategy. The goal is fee capture on a pool where the team is the natural LP.

### Optional depositor vault

When `vault.enabled: true`, the bot additionally operates as a shared vault that depositors interact with over Telegram:

- **Enrollment:** each new depositor accepts a disclaimer, provisions a per-user deposit address (keypair generated by the bot, encrypted at rest with the master key), and enrolls TOTP 2FA (Google Auth / Authy).
- **Deposits:** depositors send SOL or BERT to their per-user address. A `DepositWatcher` polls the chain for inbound transactions; a `Sweeper` consolidates funds into the main hot wallet; the `CreditEngine` converts the inbound USD value into vault shares at the current NAV per share. Every mutation is wrapped in a single SQLite transaction with an audit-log write.
- **Withdrawals:** depositors queue a USD or percentage-based withdrawal to a **pre-set whitelist address** (one whitelist per user, 7-day cooldown on changes after the first). An executor partial-closes the bot's LP position for enough SOL/BERT, routes the payout through the whitelist, and marks the row complete. Fail-closed: if the partial-close falls short, the withdrawal is refused rather than dipped from a different reserve.
- **NAV snapshots:** every tick persists a `(totalUsd, totalShares, navPerShare)` row so historical NAV is queryable and every share-count mutation has a verified price point.
- **Rate-limited privileged actions:** every privileged command (deposit-address reveal, balance, withdraw, whitelist change) requires a fresh TOTP code. Five failures in 15 minutes locks the user for 15 minutes; all lockouts are audited.
- **Operator control:** a dedicated Telegram user id (`operatorTelegramId`) can pause vault acceptance, force-process stuck items, recredit a failed deposit, or reset a user's TOTP — all audited.

### Telegram UI

When `vault.uiButtons: true` (default), every depositor flow is also available as inline-keyboard buttons. The `/menu` command opens the main keyboard (`Deposit / Balance / Withdraw / Settings / Stats`); each action walks the user through a short guided flow instead of requiring them to remember typed-command syntax. Typed commands still work unchanged — the buttons are a thin front that invokes the same handlers.

TOTP codes remain user-typed (no button can produce a one-time code). Whitelist addresses are still pasted as text. Everything else is driven by taps.

## Architecture

```
main.ts (30s poll loop)
  ├── priceOracle ─────────► Jupiter /quote + DexScreener
  │   (2-source consensus, divergence check, stale-price veto)
  ├── meteoraClient ───────► Meteora DLMM SDK + Solana RPC
  │   (refetch pool state, get active bin, open/close/swap tx builders)
  ├── strategy ────────────► in-range check, sustained-out detector
  ├── rebalancer ──────────► 9-step: drawdown → close → swap 50/50 → open → persist → notify
  ├── txSubmitter ─────────► priority fee, retry, confirm (optionally MEV-protected via Jito)
  ├── stateStore ──────────► SQLite: positions, rebalances, NAV snapshots, degraded flag
  ├── notifier ────────────► Telegram alerts (info + critical + hourly status)
  └── vault/* (optional) ──► depositorStore, enrollment, cooldowns, creditEngine,
                              depositWatcher, sweeper, withdrawalBuilder/Executor,
                              commands, operatorCommands, uiCallbacks, uiKeyboards
```

The venue surface is isolated behind `VenueClient` (see `src/venueClient.ts`) so the MM loop doesn't know DLMM-specific details — it just gets `getPosition`, `buildOpenPositionTx`, `buildClosePositionTx`, `buildSwapToRatioTx`, `getActiveBin`, `getPoolState`, `getWalletBalances`.

**Meteora DLMM specifics:**
- Positions are bin-range keypair accounts (the position pubkey is stored in `state.db` under the `nftMint` column for backward compatibility).
- The first position on a new pool costs ~0.74 SOL in bin-array rent, refunded when the position closes.
- Each bin at `bin_step=10` represents a 0.10% price step. Maximum ~70 bins per position, so `rangeWidthPct` tops out around 7%.
- `@meteora-ag/dlmm` is loaded via a `createRequire` shim in `src/meteoraClient.ts` (the package's ESM entry doesn't resolve under strict TS module mode).

## Safety model

Everything below is enforced before the rebalancer touches the chain. The bot is designed to refuse rather than misbehave.

- **`maxPositionUsd`:** hard cap on position size (e.g. $200 for a canary, $2200 for the full deploy).
- **Drawdown breaker:** compares the simulated close value against a recent baseline; if the fall exceeds `maxDrawdownPct` over `drawdownWindowMin`, the tick is refused. Fails *closed* — if `simulateClose` throws, the rebalance is refused rather than assumed safe.
- **Kill switches:** `enabled: false` in config, a file at `killSwitchFilePath`, or `systemctl stop bert-mm-bot`. The `enabled` flag is re-read every tick.
- **Initial-open guards:** the initial position open respects both the kill switch and the degraded flag.
- **Duplicate-position protection:** a failed `getPosition` RPC skips the tick (treats as "position exists, just can't read it right now") instead of assuming empty and opening a second position.
- **SOL floor:** `minSolFloorLamports` is reserved for gas before every rebalance.
- **Daily cap:** `maxRebalancesPerDay` enforced in both `strategy.decide` and the rebalancer.
- **Oracle hygiene:** `solUsd=0` samples are filtered from the mean; quotes older than `oracleStaleMinutes` are ignored.
- **Vault invariants (when enabled):**
  - Every share-count mutation wraps NAV snapshot + audit-log write in a single `DepositorStore.withTransaction` — atomic or nothing.
  - TOTP-gated commands route through a single `verifyTotpGated` choke point; a single `TotpRateLimiter` backs the lockout.
  - Sensitive flows are fail-closed: oracle null → defer; partial-close short → refuse withdrawal; `tx_sig` populated on a failed row → refuse requeue; preflight ATA failure → refuse vault enable.
  - Operator gating is by `msg.userId === cfg.vault.operatorTelegramId` (Telegram *user id*, not chat id — works in both DMs and group chats).
  - `vault-bootstrap` is run once at deploy time to create the operator user row and seed the opening NAV (1 share = $1).

## Telegram interface

The bot exposes two audiences through one chat: **depositors** (the usual users) and **operators** (Telegram user ids that match `vault.operatorTelegramId`). Both tap or type.

### Depositor commands (also reachable via the `/menu` button grid)

| Command | What it does |
|---|---|
| `/menu` | Open the main keyboard. Welcome + Create-account for non-enrolled users; main menu for enrolled. |
| `/account` | First-time: accept disclaimer → scan TOTP QR → enter code. After enrollment: show main menu. |
| `/accept`, `/decline` | Consume the disclaimer. (Equivalent to tapping `[I accept]` / `[Decline]`.) |
| `/deposit` | After TOTP, reveal the depositor's per-user deposit address. |
| `/balance` | After TOTP, show current share count + USD equivalent at live NAV. |
| `/withdraw [usd\|pct%]` | Queue a withdrawal. Button flow offers 25 / 50 / 75 / 100 % and a custom-USD entry. TOTP-gated. |
| `/setwhitelist <addr>` | Set (first-time) or change (after cooldown) the withdrawal destination. TOTP-gated. |
| `/cancelwhitelist` | Cancel a pending whitelist change. TOTP-gated. |
| `/stats` | Public vault stats: TVL + NAV/share + 24 h delta. |

### Operator commands

| Command | What it does |
|---|---|
| `/status`, `/pause`, `/resume` | MM-bot level: show status, halt rebalancing, resume. |
| `/help` | Short command summary. |
| `/pausevault`, `/resumevault` | Vault level: refuse new deposits / withdrawals (queue stays intact), or resume. |
| `/vaultstatus` | Vault TVL, pending items, lockouts, recent sweep failures. |
| `/forceprocess <id>` | Force-process a queued deposit or withdrawal after manual review. |
| `/recreditdeposit <tx_sig>` | Recredit a deposit that was detected but failed to mint shares (e.g. RPC hiccup). |
| `/resettotp <telegramId>` | Reset a user's TOTP after documented out-of-band verification. |

All operator commands auth on the caller's Telegram **user id**, so listing them publicly (via BotFather `/setcommands`) is safe — unauthorized callers receive a silent ignore.

### Inline-button UX

The UI is additive: typing works, tapping works. Replies carry contextual next-action buttons (after `Deposit` → `[Balance] [Menu]`; after `Balance` → `[Withdraw] [Deposit more] [Menu]`). Mid-flow replies carry `[Cancel]`; error replies carry `[Try again] [Menu]`. See `docs/superpowers/specs/2026-04-18-telegram-ui-design.md` for the full flow catalog.

To disable the buttons and run command-only, set `vault.uiButtons: false` and restart — callback polling stops and replies no longer carry keyboards.

## Operator CLI

Run from the project root (`/opt/bert-mm-bot` or the active worktree):

```bash
node dist/cli/index.js status              # current bot + position state
node dist/cli/index.js emergency-exit      # close the position immediately (type "CLOSE" to confirm)
node dist/cli/index.js rebalance --force   # force a rebalance regardless of strategy decision
node dist/cli/index.js clear-degraded      # clear the degraded flag after manual remediation
node dist/cli/index.js report --days 7     # PnL + time-in-range report
node dist/cli/index.js collect-fees        # collect LP fees on demand
node dist/cli/index.js close-orphan <mint> # rescue an orphaned position by pubkey
node dist/cli/index.js reconcile           # reconcile state.db against on-chain position
```

Vault bootstrap + backups have their own binaries (`dist/cli/vault-bootstrap.js`, systemd timer for DB backup).

## Deployment

See `docs/deploy-vault-v1.md` for the end-to-end runbook. The short version:

1. Build: `pnpm install && pnpm build`.
2. Fund the hot wallet. First DLMM position costs ~0.74 SOL in bin-array rent; keep a similar buffer on top for gas.
3. Start MM-only once to confirm the bot can open an initial position: `vault.enabled: false`, `systemctl start bert-mm-bot`, watch the logs and Telegram until you see `initial open complete`.
4. Stop the service. Generate + install `VAULT_MASTER_KEY`. Fill in the vault config block (remember to set `operatorTelegramId`). Run `node dist/cli/vault-bootstrap.js --initial-nav-usd <current-pool-usd>`.
5. Flip `vault.enabled: true`, restart.
6. Operator enrolls TOTP in Telegram: `/account` → `/accept` → scan QR → reply with the 6-digit code.
7. Install the backup timer (`systemd/bert-mm-bot-backup.{service,timer}`) to snapshot `state.db` on a schedule.
8. Smoke-test with a small depositor before inviting others.

## Observability

- **Logs:** JSON (`pino`) to `/var/log/bert-mm-bot/bot.log`. Every tick emits a decision line; every rebalance emits a structured timing + result line.
- **Heartbeat:** `/var/lib/bert-mm-bot/heartbeat` is touched each tick — alarm if it's stale by more than a couple of poll intervals.
- **Telegram alerts:** startup notice, initial position open, every rebalance, drawdown-breaker trip, hourly status digest (price, range, in-range %, rebalances today, bot status). The `CRITICAL` channel gets its own chat id so you can page separately.
- **Audit log (vault):** every disclaimer acceptance, TOTP enrollment, TOTP success/failure, lockout, balance reveal, withdrawal queued / processed / failed, whitelist change, operator action. Queryable from `state.db`.
- **Report CLI:** `node dist/cli/index.js report --days 7` prints a PnL + time-in-range table.
